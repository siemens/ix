# Breaking Changes V5

This document lists breaking changes introduced in Siemens Industrial Experience V5.

## Button components: accessible name on the host

Legacy properties that set an accessible name on the inner native `<button>` have been removed. Set the standard `aria-label` attribute on the custom element host instead; it is forwarded to the inner control for assistive technologies.

### ix-button

**Removed:** `ariaLabelButton` (attribute `aria-label-button`)

**Before:**

```html
<ix-button aria-label-button="Save"></ix-button>
```

**After:**

```html
<ix-button aria-label="Save"></ix-button>
```

### ix-icon-button

**Removed:** `a11yLabel` (attribute `a11y-label`)

**Before:**

```html
<ix-icon-button icon="star" a11y-label="Favorite"></ix-icon-button>
```

**After:**

```html
<ix-icon-button icon="star" aria-label="Favorite"></ix-icon-button>
```

### ix-toggle-button

**Removed:** `ariaLabelButton` (attribute `aria-label-button`)

**Before:**

```html
<ix-toggle-button aria-label-button="Bold" icon="text-bold"></ix-toggle-button>
```

**After:**

```html
<ix-toggle-button aria-label="Bold" icon="text-bold"></ix-toggle-button>
```

### ix-icon-toggle-button

**Removed:** `ariaLabelIconButton` (attribute `aria-label-icon-button`)

**Before:**

```html
<ix-icon-toggle-button icon="play" aria-label-icon-button="Play"></ix-icon-toggle-button>
```

**After:**

```html
<ix-icon-toggle-button icon="play" aria-label="Play"></ix-icon-toggle-button>
```

## Remove bootstrap font variable

Remove remaining bootstrap variable `--bs-font-sans-serif` from the styles. If you still rely on bootstrap set the font family variable via:

```css
--bs-font-sans-serif: var(--theme-font-family);
```

## Theming migration

Use this section as the single source of truth for all V5 theming updates.

### Canonical model

- Legacy theme classes such as `theme-classic-light` and `theme-classic-dark` are removed.
- IX now reads and writes theming on `<html>` only.
- A valid theme is the pair `data-ix-theme="<theme>"` and `data-ix-color-schema="<schema>"`.
- Supported color schema values are `light`, `dark`, and `system`.
- IX does not inject a default theme anymore. If your app requires one, set both attributes explicitly.

### Global rewrite rules

1. Remove all theme classes in the form `theme-<theme>-<schema>` from `<html>` and `<body>`.
2. Convert each legacy class `theme-<theme>-<schema>` to:

```html
<html data-ix-theme="<theme>" data-ix-color-schema="<schema>"></html>
```

3. Never set only one of `data-ix-theme` or `data-ix-color-schema`. Set both together.
4. Treat `theme`, `colorSchema`, and `mode` as separate concepts:
   `theme` is the theme name such as `classic`.
   `colorSchema` is the configured schema such as `light`, `dark`, or `system`.
   `mode` is the resolved runtime result and is only `light` or `dark`.

### Global before / after

```html
<html class="theme-classic-light"></html>
```

after:

```html
<html data-ix-theme="classic" data-ix-color-schema="light"></html>
```

## `ThemeSwitcher`

`ThemeSwitcher` is now attribute-based only. Remove all legacy class-based integrations.

| v4 API or behavior                                  | V5 replacement                                                            |
| --------------------------------------------------- | ------------------------------------------------------------------------- |
| `setTheme('theme-classic-dark')`                    | `setTheme('classic', 'dark')`                                             |
| `setTheme('classic', true)`                         | `setTheme('classic', 'system')`                                           |
| `setVariant(schema)`                                | `setColorSchema(schema)`                                                  |
| `getCurrentTheme()`                                 | `getTheme()` plus `getColorSchema()`                                      |
| `schemaChanged`                                     | `themeChanged`                                                            |
| `themeChanged: TypedEvent<string>`                  | `themeChanged: TypedEvent<{ theme, colorSchema, mode, isMediaChange }>`   |
| `IxTheme`                                           | Use plain theme names such as `classic` and `ThemeVariant` for the schema |
| `hasVariantSuffix()` and other legacy class helpers | Remove this logic                                                         |

### `ThemeSwitcher` before

```ts
themeSwitcher.setTheme('theme-classic-dark');
themeSwitcher.setTheme('classic', true);
themeSwitcher.setVariant('light');

themeSwitcher.themeChanged.on((theme) => {
  console.log(theme);
});

themeSwitcher.schemaChanged.on((schema) => {
  console.log(schema);
});
```

### `ThemeSwitcher` after

```ts
themeSwitcher.setTheme('classic', 'dark');
themeSwitcher.setTheme('classic', 'system');
themeSwitcher.setColorSchema('light');

themeSwitcher.themeChanged.on(({ theme, colorSchema, mode, isMediaChange }) => {
  console.log(theme, colorSchema, mode, isMediaChange);
});
```

### `ThemeSwitcher` migration rules

1. Do not pass legacy theme classes into `ThemeSwitcher`.
2. Replace every `setVariant()` call with `setColorSchema()`.
3. Replace every `schemaChanged` subscription with `themeChanged`.
4. Update every `themeChanged` handler to read the new object payload.
5. Replace every `getCurrentTheme()` usage with `getTheme()` and `getColorSchema()`.

## `ix-application`

`ix-application` now follows the same split model as the global theming API: `theme` contains only the theme name and `colorSchema` contains the schema.

| v4 API or behavior      | V5 replacement                                     |
| ----------------------- | -------------------------------------------------- |
| `theme="classic-dark"`  | `theme="classic" colorSchema="dark"`               |
| `theme="classic-light"` | `theme="classic" colorSchema="light"`              |
| `themeSystemAppearance` | `colorSchema="system"`                             |
| `theme?: IxTheme`       | `theme?: string` plus `colorSchema?: ThemeVariant` |

### `ix-application` before

```tsx
<ix-application theme="classic-dark"></ix-application>
<ix-application theme="classic" themeSystemAppearance></ix-application>
```

### `ix-application` after

```tsx
<ix-application theme="classic" colorSchema="dark"></ix-application>
<ix-application theme="classic" colorSchema="system"></ix-application>
```

### `ix-application` migration rules

1. If `theme` contains both theme and schema, split it into `theme` and `colorSchema`.
2. Replace `themeSystemAppearance` with `colorSchema="system"`.
3. Do not keep combined values such as `classic-dark` or `classic-light` in `theme`.

## `ix-toast-container`

`ix-toast-container` no longer creates a separate body-level wrapper element. The host element is the container.

| v4 API or behavior                                            | V5 replacement                             |
| ------------------------------------------------------------- | ------------------------------------------ |
| `containerId`                                                 | Set `id` on `ix-toast-container` itself    |
| `containerClass`                                              | Set `class` on `ix-toast-container` itself |
| `document.getElementById(containerId)` for the generated node | Query `ix-toast-container` directly        |
| Styling the generated wrapper                                 | Style the `ix-toast-container` host        |

### `ix-toast-container` before

```html
<ix-toast-container container-id="my-toast-container" container-class="my-toast-container" position="top-right"></ix-toast-container>
```

```ts
const container = document.getElementById('my-toast-container');
```

### `ix-toast-container` after

```html
<ix-toast-container id="my-toast-container" class="my-toast-container" position="top-right"></ix-toast-container>
```

```ts
const container = document.querySelector('ix-toast-container#my-toast-container');
```

### `ix-toast-container` migration rules

1. Remove all `containerId` and `containerClass` props.
2. Move any selector, test, or style that targeted the generated wrapper to the `ix-toast-container` host element.
3. Do not expect `showToast()` to create a second element in `document.body`.

## `ix-modal`: `disableEscapeClose` removed

Escape and the `<dialog>` **cancel** path always call **`dismissModal()`**. To block dismissal (keyboard, close control, backdrop where applicable), use **`beforeDismiss`** and return `false`.

| v4                                            | V5 replacement                                                                                                 |
| --------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| `disableEscapeClose` / `disable-escape-close` | `beforeDismiss={() => false}` (or async) when all dismissals must be vetoed                                    |
| Loading overlay not dismissible by Escape     | `showModalLoading` uses `beforeDismiss: () => false` internally; **`closeModal`** from `finish()` is unchanged |

**Note:** `beforeDismiss` runs for **every** `dismissModal` attempt, not only Escape. If you only need to block some dismiss sources, pass a **reason** (when supported) or split flows between **`closeModal`** vs **`dismissModal`**.

## Validation checklist

1. There are no remaining `theme-<theme>-<schema>` classes in application code.
2. Every `data-ix-theme` usage is paired with `data-ix-color-schema`.
3. There are no remaining `themeSystemAppearance`, `setVariant`, `getCurrentTheme`, `schemaChanged`, `containerId`, or `containerClass` usages.
4. Every `themeChanged` subscriber reads the new object payload.
5. Every `ix-application` usage passes a plain theme name, not a combined theme-and-schema value.
6. Button-related hosts use native `aria-label` instead of `aria-label-button`, `a11y-label`, or `aria-label-icon-button`.
7. There are no remaining `disableEscapeClose` / `disable-escape-close` usages; use `beforeDismiss` instead.
