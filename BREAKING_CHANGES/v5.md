# Breaking Changes V5

This document lists breaking changes introduced in Siemens Industrial Experience V5.

## Button components: accessible name on the host

Legacy properties that set an accessible name on the inner native `<button>` have been removed. Set the standard `aria-label` attribute on the custom element host instead; it is forwarded to the inner control for assistive technologies.

### ix-button

**Removed:** `ariaLabelButton` (attribute `aria-label-button`)

**Before:**

```html
<ix-button aria-label-button="Save"></ix-button>
```

**After:**

```html
<ix-button aria-label="Save"></ix-button>
```

### ix-icon-button

**Removed:** `a11yLabel` (attribute `a11y-label`)

**Before:**

```html
<ix-icon-button icon="star" a11y-label="Favorite"></ix-icon-button>
```

**After:**

```html
<ix-icon-button icon="star" aria-label="Favorite"></ix-icon-button>
```

### ix-toggle-button

**Removed:** `ariaLabelButton` (attribute `aria-label-button`)

**Before:**

```html
<ix-toggle-button aria-label-button="Bold" icon="text-bold"></ix-toggle-button>
```

**After:**

```html
<ix-toggle-button aria-label="Bold" icon="text-bold"></ix-toggle-button>
```

### ix-icon-toggle-button

**Removed:** `ariaLabelIconButton` (attribute `aria-label-icon-button`)

**Before:**

```html
<ix-icon-toggle-button icon="play" aria-label-icon-button="Play"></ix-icon-toggle-button>
```

**After:**

```html
<ix-icon-toggle-button icon="play" aria-label="Play"></ix-icon-toggle-button>
```

## Remove bootstrap font variable

Remove remaining bootstrap variable `--bs-font-sans-serif` from the styles. If you still rely on bootstrap set the font family variable via:

```css
--bs-font-sans-serif: var(--theme-font-family);
```

## `ix-time-picker` hour column header attribute

**Removed:** `i18n-column-header`

**Added:** `i18n-hour-column-header` (aligned with `ix-time-input` and the other time-column header attributes).

**Before:**

```html
<ix-time-picker i18n-column-header="Hr"></ix-time-picker>
```

**After:**

```html
<ix-time-picker i18n-hour-column-header="Hr"></ix-time-picker>
```

The JavaScript property name (`i18nHourColumnHeader`) is unchanged.

## Theming migration

Use this section as the single source of truth for all V5 theming updates.

### Canonical model

- Legacy theme classes such as `theme-classic-light` and `theme-classic-dark` are removed.
- IX now reads and writes theming on `<html>` only.
- A valid theme is the pair `data-ix-theme="<theme>"` and `data-ix-color-schema="<schema>"`.
- Supported color schema values are `light`, `dark`, and `system`.
- IX does not inject a default theme anymore. If your app requires one, set both attributes explicitly.

### Global rewrite rules

1. Remove all theme classes in the form `theme-<theme>-<schema>` from `<html>` and `<body>`.
2. Convert each legacy class `theme-<theme>-<schema>` to:

```html
<html data-ix-theme="<theme>" data-ix-color-schema="<schema>"></html>
```

3. Never set only one of `data-ix-theme` or `data-ix-color-schema`. Set both together.
4. Treat `theme`, `colorSchema`, and `mode` as separate concepts:
   `theme` is the theme name such as `classic`.
   `colorSchema` is the configured schema such as `light`, `dark`, or `system`.
   `mode` is the resolved runtime result and is only `light` or `dark`.

### Global before / after

```html
<html class="theme-classic-light"></html>
```

after:

```html
<html data-ix-theme="classic" data-ix-color-schema="light"></html>
```

### `ThemeSwitcher`

`ThemeSwitcher` is now attribute-based only. Remove all legacy class-based integrations.

| v4 API or behavior                                  | V5 replacement                                                            |
| --------------------------------------------------- | ------------------------------------------------------------------------- |
| `setTheme('theme-classic-dark')`                    | `setTheme('classic', 'dark')`                                             |
| `setTheme('classic', true)`                         | `setTheme('classic', 'system')`                                           |
| `setVariant(schema)`                                | `setColorSchema(schema)`                                                  |
| `getCurrentTheme()`                                 | `getTheme()` plus `getColorSchema()`                                      |
| `schemaChanged`                                     | `themeChanged`                                                            |
| `themeChanged: TypedEvent<string>`                  | `themeChanged: TypedEvent<{ theme, colorSchema, mode, isMediaChange }>`   |
| `IxTheme`                                           | Use plain theme names such as `classic` and `ThemeVariant` for the schema |
| `hasVariantSuffix()` and other legacy class helpers | Remove this logic                                                         |

#### `ThemeSwitcher` before

```ts
themeSwitcher.setTheme('theme-classic-dark');
themeSwitcher.setTheme('classic', true);
themeSwitcher.setVariant('light');

themeSwitcher.themeChanged.on((theme) => {
  console.log(theme);
});

themeSwitcher.schemaChanged.on((schema) => {
  console.log(schema);
});
```

#### `ThemeSwitcher` after

```ts
themeSwitcher.setTheme('classic', 'dark');
themeSwitcher.setTheme('classic', 'system');
themeSwitcher.setColorSchema('light');

themeSwitcher.themeChanged.on(({ theme, colorSchema, mode, isMediaChange }) => {
  console.log(theme, colorSchema, mode, isMediaChange);
});
```

#### `ThemeSwitcher` migration rules

1. Do not pass legacy theme classes into `ThemeSwitcher`.
2. Replace every `setVariant()` call with `setColorSchema()`.
3. Replace every `schemaChanged` subscription with `themeChanged`.
4. Update every `themeChanged` handler to read the new object payload.
5. Replace every `getCurrentTheme()` usage with `getTheme()` and `getColorSchema()`.

### `ix-application`

`ix-application` now follows the same split model as the global theming API: `theme` contains only the theme name and `colorSchema` contains the schema.

| v4 API or behavior      | V5 replacement                                     |
| ----------------------- | -------------------------------------------------- |
| `theme="classic-dark"`  | `theme="classic" colorSchema="dark"`               |
| `theme="classic-light"` | `theme="classic" colorSchema="light"`              |
| `themeSystemAppearance` | `colorSchema="system"`                             |
| `theme?: IxTheme`       | `theme?: string` plus `colorSchema?: ThemeVariant` |

#### `ix-application` before

```tsx
<ix-application theme="classic-dark"></ix-application>
<ix-application theme="classic" themeSystemAppearance></ix-application>
```

#### `ix-application` after

```tsx
<ix-application theme="classic" colorSchema="dark"></ix-application>
<ix-application theme="classic" colorSchema="system"></ix-application>
```

#### `ix-application` migration rules

1. If `theme` contains both theme and schema, split it into `theme` and `colorSchema`.
2. Replace `themeSystemAppearance` with `colorSchema="system"`.
3. Do not keep combined values such as `classic-dark` or `classic-light` in `theme`.

### Validation checklist

1. There are no remaining `theme-<theme>-<schema>` classes in application code.
2. Every `data-ix-theme` usage is paired with `data-ix-color-schema`.
3. There are no remaining `themeSystemAppearance`, `setVariant`, `getCurrentTheme`, `schemaChanged`, `containerId`, or `containerClass` usages.
4. Every `themeChanged` subscriber reads the new object payload.
5. Every `ix-application` usage passes a plain theme name, not a combined theme-and-schema value.

## `ix-toast-container`

`ix-toast-container` no longer creates a separate body-level wrapper element. The host element is the container.

| v4 API or behavior                                            | V5 replacement                             |
| ------------------------------------------------------------- | ------------------------------------------ |
| `containerId`                                                 | Set `id` on `ix-toast-container` itself    |
| `containerClass`                                              | Set `class` on `ix-toast-container` itself |
| `document.getElementById(containerId)` for the generated node | Query `ix-toast-container` directly        |
| Styling the generated wrapper                                 | Style the `ix-toast-container` host        |

### `ix-toast-container` before

```html
<ix-toast-container container-id="my-toast-container" container-class="my-toast-container" position="top-right"></ix-toast-container>
```

```ts
const container = document.getElementById('my-toast-container');
```

### `ix-toast-container` after

```html
<ix-toast-container id="my-toast-container" class="my-toast-container" position="top-right"></ix-toast-container>
```

```ts
const container = document.querySelector('ix-toast-container#my-toast-container');
```

### `ix-toast-container` migration rules

1. Remove all `containerId` and `containerClass` props.
2. Move any selector, test, or style that targeted the generated wrapper to the `ix-toast-container` host element.
3. Do not expect `showToast()` to create a second element in `document.body`.

## `ix-modal`: `disableEscapeClose` removed

Escape and the `<dialog>` **cancel** path always call **`dismissModal()`**. To block dismissal (keyboard, close control, backdrop where applicable), use **`beforeDismiss`** and return `false`.

| v4                                            | V5 replacement                                                                                                 |
| --------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| `disableEscapeClose` / `disable-escape-close` | `beforeDismiss={() => false}` (or async) when all dismissals must be vetoed                                    |
| Loading overlay not dismissible by Escape     | `showModalLoading` uses `beforeDismiss: () => false` internally; **`closeModal`** from `finish()` is unchanged |

**Note:** `beforeDismiss` runs for **every** `dismissModal` attempt, not only Escape. If you only need to block some dismiss sources, pass a **reason** (when supported) or split flows between **`closeModal`** vs **`dismissModal`**.

## `ix-aggrid` requires a direct `@siemens/ix` dependency

`@siemens/ix-aggrid` now declares `@siemens/ix` as a peer dependency instead of installing it transitively. Applications that use the AG Grid theme must add `@siemens/ix` to their own dependencies so IX CSS imports and related assets resolve explicitly from the app.

### Before

```json
{
  "dependencies": {
    "@siemens/ix-aggrid": "^4.0.0"
  }
}
```

### After

```json
{
  "dependencies": {
    "@siemens/ix": "^4.4.0",
    "@siemens/ix-aggrid": "^5.0.0"
  }
}
```

### Migration rules

1. Add `@siemens/ix` to the application package that consumes `@siemens/ix-aggrid`.
2. Keep the installed `@siemens/ix` version within the peer dependency range required by `@siemens/ix-aggrid`.
3. Continue importing the required IX CSS from the application entry point before applying the AG Grid theme.

### Validation checks

1. Every application using `@siemens/ix-aggrid` declares `@siemens/ix` directly.
2. The installed `@siemens/ix` version satisfies the `@siemens/ix-aggrid` peer dependency range.
3. IX CSS imports still come from the consuming application, not from `@siemens/ix-aggrid`.

## `ix-split-button`: `placement` removed

The `placement` property on `ix-split-button` was not applied to the internal menu. It is removed as a public API to avoid a misleading no-op.

| v4 API or behavior        | V5 replacement |
| ------------------------- | -------------- |
| `placement` / `placement` | Remove         |

### `ix-split-button` before

```html
<ix-split-button label="Save" placement="bottom-start">
  <ix-dropdown-item label="Item 1"></ix-dropdown-item>
</ix-split-button>
```

### `ix-split-button` after

```html
<ix-split-button label="Save">
  <ix-dropdown-item label="Item 1"></ix-dropdown-item>
</ix-split-button>
```

### `ix-split-button` migration rules

1. Remove every `placement` (and `placement` attribute) from `ix-split-button`.
2. No layout change is expected: the v4 value was not used when positioning the dropdown.

## ix-tabs: Key-based tab identification

The tab selection model changed from index-based (`selected` prop) to key-based (`activeTabKey` + `tabKey`).

### Removed props

| Component | Removed                           | Replacement                                           |
| --------- | --------------------------------- | ----------------------------------------------------- |
| `ix-tabs` | `selected` (number)               | `activeTabKey` (string)                               |
| `ix-tabs` | `ariaLabelChevronLeftIconButton`  | Removed (scroll arrows replaced by overflow dropdown) |
| `ix-tabs` | `ariaLabelChevronRightIconButton` | Removed                                               |

### Removed events

| Component | Removed                         | Replacement                       |
| --------- | ------------------------------- | --------------------------------- |
| `ix-tabs` | `selectedChange` (emits number) | `tabChange` (emits tabKey string) |

### New required props

| Component     | Prop     | Description                           |
| ------------- | -------- | ------------------------------------- |
| `ix-tab-item` | `tabKey` | Unique string key identifying the tab |

### New optional props

| Component     | Prop       | Description                            |
| ------------- | ---------- | -------------------------------------- |
| `ix-tab-item` | `label`    | Tab text (preferred over slot content) |
| `ix-tab-item` | `icon`     | Icon name for the tab                  |
| `ix-tab-item` | `closable` | Enables close button on the tab        |

### Internal props (no longer consumer-settable)

`ix-tab-item` props `small`, `layout`, `placement`, and `rounded` are now set automatically by the parent `ix-tabs`. Remove any direct usage of these props on `ix-tab-item`.

### Migration

```html
<!-- Before -->
<ix-tabs selected="1">
  <ix-tab-item>Tab 1</ix-tab-item>
  <ix-tab-item>Tab 2</ix-tab-item>
</ix-tabs>

<!-- After -->
<ix-tabs active-tab-key="tab-2">
  <ix-tab-item tab-key="tab-1" label="Tab 1"></ix-tab-item>
  <ix-tab-item tab-key="tab-2" label="Tab 2"></ix-tab-item>
</ix-tabs>
```

Replace event listeners:

```js
// Before
tabs.addEventListener('selectedChange', (e) => handleTab(e.detail)); // index: 0, 1, 2

// After
tabs.addEventListener('tabChange', (e) => handleTab(e.detail)); // key: 'tab-1', 'tab-2'
```

### Validation checklist

1. No remaining `selected` prop usage on `ix-tabs`.
2. Every `ix-tab-item` has a unique `tab-key` attribute.
3. All `selectedChange` listeners replaced with `tabChange`.
4. No direct `small`, `layout`, `placement`, or `rounded` props on `ix-tab-item`.

## ix-menu-about and ix-menu-settings: Slotted `ix-tabs` opt-in via `suppressLegacyTabs`

`ix-menu-about` and `ix-menu-settings` keep their legacy `ix-menu-about-item` / `ix-menu-settings-item` rendering by default. To adopt the slotted `ix-tabs` / `ix-tab-item` pattern, set `suppressLegacyTabs` on the parent component and manage the visible content in your application.

### Renamed props

| Component       | Old Prop                  | New Prop                             |
| --------------- | ------------------------- | ------------------------------------ |
| `ix-menu-about` | `activeTabLabel` (string) | `activeTabKey` (string, legacy mode) |

### New opt-in props

| Component          | Prop                 | Description                                      |
| ------------------ | -------------------- | ------------------------------------------------ |
| `ix-menu-about`    | `suppressLegacyTabs` | Opts into the slotted `ix-tabs` integration path |
| `ix-menu-settings` | `suppressLegacyTabs` | Opts into the slotted `ix-tabs` integration path |

### New required props

| Component               | Prop     | Description                           |
| ----------------------- | -------- | ------------------------------------- |
| `ix-menu-about-item`    | `tabKey` | Unique string key identifying the tab |
| `ix-menu-settings-item` | `tabKey` | Unique string key identifying the tab |

### ix-menu-about-news

| Old Prop                                    | New Prop            | Description                                                                |
| ------------------------------------------- | ------------------- | -------------------------------------------------------------------------- |
| `aboutItemLabel` (used as footer condition) | `activeAboutTabKey` | Controls "Show more" button visibility and navigates to specific about tab |

### Shadow DOM change

`ix-menu-about-item` changed from `shadow: true` to `shadow: false`. If you targeted internal elements via `::part()` or shadow DOM queries, update to plain CSS selectors.

### Migration

```html
<!-- Before -->
<ix-menu-about active-tab-label="Tab 1">
  <ix-menu-about-item label="Tab 1">Content 1</ix-menu-about-item>
  <ix-menu-about-item label="Tab 2">Content 2</ix-menu-about-item>
</ix-menu-about>

<!-- After (slotted ix-tabs path) -->
<ix-menu-about suppress-legacy-tabs>
  <ix-tabs active-tab-key="tab-1">
    <ix-tab-item tab-key="tab-1">Tab 1</ix-tab-item>
    <ix-tab-item tab-key="tab-2">Tab 2</ix-tab-item>
  </ix-tabs>
  <section role="tabpanel" data-tab-content="tab-1">Content 1</section>
  <section role="tabpanel" data-tab-content="tab-2" hidden>Content 2</section>
</ix-menu-about>

<script>
  const about = document.querySelector('ix-menu-about');
  const tabs = about.querySelector('ix-tabs');

  const updateVisibleContent = (tabKey) => {
    about.querySelectorAll('[data-tab-content]').forEach((content) => {
      content.hidden = content.dataset.tabContent !== tabKey;
    });
  };

  updateVisibleContent(tabs.activeTabKey ?? 'tab-1');
  tabs.addEventListener('tabChange', (event) => {
    updateVisibleContent(event.detail ?? 'tab-1');
  });
</script>
```

```html
<!-- Apply the same pattern to settings -->
<ix-menu-settings suppress-legacy-tabs>
  <ix-tabs active-tab-key="tab-1">
    <ix-tab-item tab-key="tab-1">Tab 1</ix-tab-item>
  </ix-tabs>
  <section role="tabpanel" data-tab-content="tab-1">Content 1</section>
</ix-menu-settings>
```

```html
<!-- Legacy item-based path -->
<ix-menu-about active-tab-key="tab-1">
  <ix-menu-about-item tab-key="tab-1" label="Tab 1">Content 1</ix-menu-about-item>
  <ix-menu-about-item tab-key="tab-2" label="Tab 2">Content 2</ix-menu-about-item>
</ix-menu-about>
```

```html
<!-- Legacy item-based path -->
<ix-menu-settings active-tab-key="tab-1">
  <ix-menu-settings-item tab-key="tab-1" label="Tab 1">Content 1</ix-menu-settings-item>
</ix-menu-settings>
```

```html
<!-- Before -->
<ix-menu-about-news about-item-label="Tab 1"> News content </ix-menu-about-news>

<!-- After -->
<ix-menu-about-news active-about-tab-key="tab-1"> News content </ix-menu-about-news>
```

### Validation checklist

1. No remaining `active-tab-label` usage on `ix-menu-about` — use `active-tab-key`.
2. Slotted `ix-tabs` integrations set `suppress-legacy-tabs` on the parent menu component.
3. New integrations use `ix-tabs` / `ix-tab-item` plus slotted content managed by the application.
4. No application code uses internal `ix-tab-panel` / `ix-tab-panels`.
5. Every `ix-menu-about-item` and `ix-menu-settings-item` has a unique `tab-key` attribute.
6. Replace `about-item-label` with `active-about-tab-key` on `ix-menu-about-news`.
